# 内容结构模板

## 概述

当用户的主题与「内容构思深坑 / 表达呈现雷区 / 项目汇报避坑」相关时，按以下结构设计 PPT。

## 结构概览

```
1. 封面页
2. 痛点故事页（Why）
3. 问题总览页
4. Part 1：内容构思的三大深坑
   4.1 深坑一：痛点不痛
   4.2 深坑二：解决方案自说自话
   4.3 深坑三：价值空洞
5. Part 2：表达呈现的四大雷区
   5.1 雷区一：PPT 当成演讲稿
   5.2 雷区二：语言技术化
   5.3 雷区三：缺乏故事线
   5.4 雷区四：演示环节薄弱
6. 总结与行动呼吁页
```

---

## 1. 封面页

### 目标

- 一句标题 + 一句副标题
- 标出场景：如「AI 编程大赛 / 校内路演 / 内部分享」

### 建议包含

| 元素     | 说明                   |
| -------- | ---------------------- |
| 演讲主题 | 简洁有力的标题         |
| 副标题   | 补充说明或受众定位     |
| 演讲者   | 姓名/团队名称（可选）  |
| 场合     | 时间或活动名称（可选） |

### slide_type

`cover`

---

## 2. 破题页：痛点故事（Why）

### 目标

用一个「**具体人物 + 具体场景 + 具体任务 + 具体痛点数据**」讲故事。

### 故事模板

```
【角色】的【场景】，需要完成【任务】，
但面临【具体困难】，导致【量化后果】。
```

### 参考示例

> 「财务部张姐，每月底需要手动核对上千张发票，眼花缭乱，导致错误率高达 5%，员工频繁投诉，财务风险增加。」

### 核心要求

- 让听众觉得「这个问题我见过 / 我也遇到过」
- 必须有具体角色、具体数据

### slide_type

`story`

---

## 3. 问题总览页

### 目标

把故事抽象为两个维度的问题地图。

### 内容

| 维度         | 说明                    |
| ------------ | ----------------------- |
| 内容构思问题 | 写什么                  |
| 表达呈现问题 | 怎么讲、怎么放在 PPT 上 |

### 结构

```
内容构思的三大深坑          表达呈现的四大雷区
├── 痛点不痛               ├── PPT当成演讲稿
├── 解决方案自说自话        ├── 语言技术化
└── 价值空洞               ├── 缺乏故事线
                          └── 演示环节薄弱
```

### slide_type

`overview`

---

## 4. Part 1：内容构思的三大深坑

### 4.1 深坑一：痛点不痛

#### 典型错误

- 只说「效率低」「体验不好」
- 只有模糊感受，没有具体场景

#### 后果

- 评委没有感觉，觉得可做可不做

#### 正确做法

讲一个「人」的故事：

| 要素           | 示例                 |
| -------------- | -------------------- |
| 谁（角色）     | 财务部张姐           |
| 在哪（场景）   | 月底结算期           |
| 做什么（任务） | 核对上千张发票       |
| 多难（数据）   | 错误率 5%，加班 3 天 |

#### speaker_notes 要求

必须给出至少一个完整的故事示例。

#### slide_type

`pitfall`

---

### 4.2 深坑二：解决方案自说自话

#### 典型错误

- 只说「我们用了 AI 大模型」「用了 xxx 技术」
- 堆技术参数，不讲业务逻辑
- 说不清 AI 助手在业务流程哪个环节发挥作用

#### 后果

- 评委不知道你究竟解决了什么

#### 正确做法：填空公式

```
在【某业务环节】，
为解决【某具体任务】，
我们用【AI/编程助手】来【执行具体动作】，
使【关键指标】从 A 提升到 B。
```

#### 填空示例

> 在【合同审查环节】，为解决【条款遗漏风险】，我们用【AI 助手】来【自动识别关键条款并标注风险】，使【审查时间】从 3 小时缩短到 10 分钟。

#### speaker_notes 要求

给出 1–2 个与当前主题贴合的填空示例。

#### slide_type

`pitfall` + `solution`（可能需要 2 页）

---

### 4.3 深坑三：价值空洞

#### 典型错误

- 「效果很好」「大大提升」这类空话
- 没有前后对比数据

#### 后果

- 成果不可衡量，可信度低

#### 正确做法

用数据说话，量化价值：

```
将【指标A】从【原值】改善到【新值】，
预计每年可【节省/创造】【具体金额】。
```

#### 量化示例

> 将合同审查时间从 3 小时缩短到 10 分钟，错误率从 5% 降到 0.1%，预计每年可节省 20 万元人力成本。

#### 专用页面

设计一页专门展示「前后对比」的 PPT（表格或对比条）。

#### slide_type

`pitfall` + `comparison`

---

## 5. Part 2：表达呈现的四大雷区

### 5.1 雷区一：PPT 当成演讲稿

#### 错误表现

- PPT 上全是文字
- 演讲者照着念

#### 后果

- 评委忙着看字，听不进人

#### 正确做法

| 原则       | 说明                   |
| ---------- | ---------------------- |
| 一页一观点 | 不要堆砌内容           |
| 多用图表   | 流程图、架构图、数据图 |
| 关键词突出 | 加粗放大核心词         |

#### 专用页面

设计「错误示例 vs 正确示例」对比结构。

#### slide_type

`pitfall` + `comparison`

---

### 5.2 雷区二：语言技术化

#### 错误表现

- 满口「Transformer」「NLU 解析」「多模态对齐」等术语

#### 后果

- 非技术评委听不懂，失去兴趣和分数

#### 正确做法

说「人话」，多打比方：

> 「我们的系统就像一个不知疲倦的超级助理，能瞬间读完上百页资料，并提炼重点。」

#### speaker_notes 要求

给出 1–2 个生动比喻。

#### slide_type

`pitfall`

---

### 5.3 雷区三：缺乏故事线

#### 错误表现

- PPT 是流水账：「第一步、第二步……」

#### 后果

- 评委记不住重点，觉得枯燥

#### 正确做法

用「黄金圈」结构组织 PPT：

```
Why（为什么做？）
  └─ 痛点故事
      ↓
How（怎么做？）
  └─ 方案与 AI 应用
      ↓
What（结果如何？）
  └─ 量化成果与展望
```

#### key_points 要求

在结构上呼应黄金圈，点明当前所处环节。

#### slide_type

`structure`

---

### 5.4 雷区四：演示环节薄弱

#### 错误表现

- 只有静态截图
- 临场演示卡顿、翻车

#### 后果

- 评委质疑项目真实性和成熟度

#### 正确做法

| 策略     | 说明                        |
| -------- | --------------------------- |
| 优先录屏 | 流畅展示，避免现场风险      |
| 聚焦核心 | 只演示 1–2 个最有价值的功能 |
| 备选方案 | 准备截图作为 Plan B         |

#### 专用页面

设计一页专门说明「演示策略」，包括准备建议。

#### slide_type

`demo`

---

## 6. 收尾页：总结与行动呼吁

### 目标

- 总结 3–5 条「今天的避坑 checklist」
- 给出「下一步行动」

### 内容模板

```
## 避坑 Checklist

□ 痛点有具体角色、场景、任务、数据
□ 方案说清"在哪个环节用AI做什么"
□ 价值有前后对比数据
□ PPT 一页一观点，多用图
□ 语言说人话，避免术语
□ 有清晰的 Why-How-What 结构
□ 演示优先录屏，聚焦核心功能
```

### 行动呼吁示例

> 「用这套模板重构你的项目路演 PPT，今晚就开始！」

### slide_type

`summary` + `cta`

---

## 页数估算

| 部分     | 预估页数     |
| -------- | ------------ |
| 封面     | 1            |
| 痛点故事 | 1            |
| 问题总览 | 1            |
| 三大深坑 | 4-6          |
| 四大雷区 | 5-7          |
| 总结行动 | 1-2          |
| **总计** | **13-18 页** |

---

## 灵活调整原则

如果用户主题不是「深坑/雷区」类：

1. **保留核心框架**：
   - 封面 → 痛点故事 → 结构总览 → 分章节 → 总结

2. **可简化模块**：
   - 删减「三大深坑 / 四大雷区」
   - 保留最通用的部分

3. **始终遵循**：
   - Why → How → What 的故事线
   - 统一的输出格式（见 `output-format.md`）
   - 统一的视觉风格（见 `visual-rules.md`）
