# Definition of Ready (DoR) - 需求准入标准

## 概述

DoR 定义了需求可以进入开发阶段的最低标准。只有满足 DoR 的需求才能被纳入规划。

## 核心检查项

### 1. 需求清晰度

| 检查项       | 标准                               | 验证方式 |
| ------------ | ---------------------------------- | -------- |
| 需求描述完整 | 有明确的功能描述和范围             | 团队评审 |
| 无歧义       | 不同人理解一致                     | 团队讨论 |
| 可测试       | 可以编写测试用例                   | QA 评审  |
| 独立性       | 可独立交付，不强依赖其他未完成需求 | 依赖分析 |

### 2. 验收标准

| 检查项 | 标准                     | 示例                   |
| ------ | ------------------------ | ---------------------- |
| 具体   | 描述具体行为而非抽象概念 | "点击登录后显示首页"   |
| 可度量 | 有明确的成功/失败判定    | "响应时间 < 200ms"     |
| 可验证 | 可以通过测试验证         | "单元测试覆盖 80%"     |
| 完整   | 覆盖正常流程和异常流程   | "密码错误显示错误提示" |

### 3. 技术可行性

| 检查项   | 标准                         |
| -------- | ---------------------------- |
| 技术方案 | 有初步的技术实现方案         |
| 风险评估 | 识别了主要技术风险           |
| 依赖确认 | 外部依赖已确认可用           |
| 资源评估 | 所需资源（人力、环境）可获得 |

### 4. 设计就绪

| 检查项   | 标准                      |
| -------- | ------------------------- |
| UI 设计  | 有 UI 设计稿（如适用）    |
| API 设计 | 有 API 接口定义（如适用） |
| 数据模型 | 数据结构已定义            |
| 交互流程 | 用户流程图已确认          |

## 检查清单模板

```markdown
## DoR 检查清单 - [需求ID]

### 需求清晰度

- [ ] 需求描述完整，包含功能、范围、边界
- [ ] 团队对需求理解一致，无歧义
- [ ] 需求可独立交付

### 验收标准

- [ ] 定义了明确的验收标准
- [ ] 验收标准可测试、可度量
- [ ] 覆盖了正常流程和异常流程

### 技术可行性

- [ ] 有初步技术方案
- [ ] 识别了技术风险和缓解措施
- [ ] 外部依赖已确认

### 设计就绪

- [ ] UI 设计已完成（如适用）
- [ ] API 接口已定义（如适用）
- [ ] 数据模型已确定

### 其他

- [ ] 优先级已确定
- [ ] 预估已完成
- [ ] 相关方已知晓

---

检查人: ****\_\_\_****
检查日期: ****\_\_\_****
结果: [ ] 通过 [ ] 不通过
```

## 常见不满足 DoR 的情况

| 问题         | 描述                    | 解决方案            |
| ------------ | ----------------------- | ------------------- |
| 需求模糊     | "优化性能" 没有具体指标 | 明确性能指标和基线  |
| 缺少验收标准 | 无法判断何时完成        | 补充具体验收标准    |
| 技术不确定   | 不知道能否实现          | 先做技术调研        |
| 设计缺失     | 没有 UI/API 设计        | 完成设计后再开发    |
| 依赖未确认   | 依赖的服务/数据未就绪   | 确认依赖或使用 Mock |
| 范围过大     | 一个需求包含太多功能    | 拆分为多个需求      |

## DoR 豁免

某些情况下可以豁免部分 DoR 检查项：

| 情况     | 可豁免项         | 条件                 |
| -------- | ---------------- | -------------------- |
| 紧急修复 | 设计就绪         | 需记录技术债务       |
| 技术探索 | 验收标准详细程度 | 有明确探索目标       |
| 原型验证 | 完整设计         | 仅用于验证，不上生产 |

豁免需要：

1. 明确记录豁免原因
2. 获得相关方同意
3. 制定补救计划
