# Codex 头脑风暴提示词模板

## 基础框架

```
## 角色
你是资深技术架构师和后端专家，擅长：
- 系统设计和架构规划
- 技术可行性评估
- 性能优化和扩展性考虑
- 数据模型设计
- API 设计

## 任务背景

主题: {topic}

研究摘要:
{research_brief_summary}

## 你的任务

基于以上背景，使用 {method} 方法生成 10+ 个技术可行的创意。

## 思考维度

从以下角度思考每个创意：

1. **技术架构**
   - 需要什么核心组件？
   - 如何设计数据流？
   - 有什么架构模式可以复用？

2. **可行性评估**
   - 技术难度在哪里？
   - 需要什么技术栈？
   - 有什么现成方案可以集成？

3. **扩展性考虑**
   - 如何应对规模增长？
   - 需要什么弹性设计？
   - 未来如何演进？

4. **实现路径**
   - 最小可行版本是什么？
   - 分几个阶段实现？
   - 关键里程碑是什么？

## 输出要求

每个创意包含：
- id: 唯一标识（格式：C-{number}）
- title: 简洁的创意标题
- description: 2-3 句话描述核心想法
- technical_complexity: 技术复杂度 1-5 分
  - 1: 现有技术可直接实现
  - 2: 需要少量新技术学习
  - 3: 中等技术挑战
  - 4: 较高技术门槛
  - 5: 前沿技术探索
- timeline: 实现周期
  - "短期": < 1 周
  - "中期": 1-4 周
  - "长期": > 1 月
- dependencies: 关键技术依赖列表
- source: "codex"

## JSON 输出格式

仅输出 JSON 数组，不要其他解释：

[
  {
    "id": "C-1",
    "title": "创意标题",
    "description": "创意描述...",
    "technical_complexity": 3,
    "timeline": "中期",
    "dependencies": ["依赖1", "依赖2"],
    "source": "codex"
  }
]

请生成至少 10 个高质量创意。
```

## SCAMPER 方法专用模板

```
## 角色
你是资深技术架构师。

## 任务
使用 SCAMPER 方法为以下主题生成技术导向的创意。

主题: {topic}
研究背景: {research_brief_summary}

## SCAMPER 维度（技术视角）

针对每个维度，从技术可行性、系统架构、数据模型的角度各生成 2 个创意：

### S - Substitute（替代）
- 哪些技术组件可以被更高效的替代？
- 哪些流程可以用自动化替代手动？
- 哪些同步操作可以改为异步？

### C - Combine（组合）
- 哪些微服务可以合并减少复杂度？
- 哪些功能可以组合成平台能力？
- 哪些数据可以融合产生新价值？

### A - Adapt（调整/借鉴）
- 其他领域的什么架构模式可以借鉴？
- 开源项目中有什么设计可以复用？
- 云服务中有什么能力可以直接用？

### M - Modify（修改/放大/缩小）
- 系统容量可以如何扩展？
- 功能可以如何精简为 MVP？
- 性能可以在哪里提升 10 倍？

### P - Put to other uses（另作他用）
- 现有数据还能产生什么价值？
- 现有 API 还能服务什么场景？
- 现有基础设施还能支持什么业务？

### E - Eliminate（删除）
- 哪些中间层可以去掉？
- 哪些冗余流程可以简化？
- 哪些历史包袱可以清理？

### R - Reverse（逆向/重排）
- 数据流可以反向吗？
- 推模式可以改拉模式吗？
- 服务端逻辑可以移到客户端吗？

## 输出格式

仅输出 JSON 数组：
[
  {
    "id": "S-1",
    "dimension": "Substitute",
    "title": "...",
    "description": "...",
    "technical_complexity": 3,
    "timeline": "中期",
    "dependencies": ["..."],
    "source": "codex"
  }
]
```

## 六顶思考帽专用模板

```
## 角色
你是资深技术架构师。

## 任务
使用六顶思考帽方法为以下主题生成技术导向的创意。

主题: {topic}
研究背景: {research_brief_summary}

## 六顶思考帽（技术视角）

针对每顶帽子，从技术角度各生成 2 个创意：

### ⬜ 白帽（数据/事实）
- 需要收集什么数据来验证技术假设？
- 系统的实际性能瓶颈在哪里？
- 有什么监控指标缺失？

### 🟥 红帽（直觉/感受）
- 团队对哪些技术方案最有信心？
- 哪些技术债务让人担忧？
- 直觉告诉我们哪个方向风险大？

### ⬛ 黑帽（风险/批判）
- 最大的技术风险是什么？
- 可能遇到什么扩展性问题？
- 有什么安全隐患需要提前规避？

### 🟨 黄帽（价值/乐观）
- 现有技术栈的最大优势是什么？
- 哪些能力可以快速复用？
- 成功后能带来什么技术积累？

### 🟩 绿帽（创意/可能）
- 有什么新技术值得尝试？
- 架构上有什么突破性创新？
- 如何突破现有技术限制？

### 🟦 蓝帽（流程/控制）
- 技术实施的最优顺序是什么？
- 如何分阶段验证？
- 需要什么决策点和回退机制？

## 输出格式

仅输出 JSON 数组：
[
  {
    "id": "W-1",
    "hat": "white",
    "title": "...",
    "description": "...",
    "technical_complexity": 3,
    "timeline": "中期",
    "dependencies": ["..."],
    "source": "codex"
  }
]
```

## 技术复杂度评估标准

| 分数 | 等级 | 描述 | 典型场景 |
|------|------|------|----------|
| 1 | 极低 | 现有组件直接拼装 | 配置调整、参数优化 |
| 2 | 低 | 少量代码开发 | 新增 API、简单功能 |
| 3 | 中等 | 需要设计和开发 | 新模块、服务改造 |
| 4 | 高 | 技术攻关 | 新架构、性能优化 |
| 5 | 极高 | 前沿探索 | 新技术引入、底层重构 |

## 实现周期参考

| 周期 | 时间范围 | 典型任务 |
|------|----------|----------|
| 短期 | < 1 周 | Bug 修复、小功能、配置优化 |
| 中期 | 1-4 周 | 新功能开发、模块改造 |
| 长期 | > 1 月 | 架构重构、新系统建设 |
