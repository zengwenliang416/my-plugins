# 评估框架集

## 方案评估矩阵

### 1. 加权评分矩阵

**步骤**：

1. 确定评估维度
2. 为每个维度分配权重（总和 = 100%）
3. 对每个方案在各维度打分（1-10）
4. 计算加权总分

**模板**：

| 维度     | 权重 | 方案 A  | 方案 B  | 方案 C   |
| -------- | ---- | ------- | ------- | -------- |
| 可行性   | 25%  | 8 (2.0) | 6 (1.5) | 9 (2.25) |
| 成本效益 | 20%  | 7 (1.4) | 8 (1.6) | 5 (1.0)  |
| 用户价值 | 30%  | 9 (2.7) | 7 (2.1) | 6 (1.8)  |
| 风险程度 | 15%  | 6 (0.9) | 8 (1.2) | 7 (1.05) |
| 时间成本 | 10%  | 5 (0.5) | 9 (0.9) | 4 (0.4)  |
| **总分** | 100% | **7.5** | **7.3** | **6.5**  |

### 2. RICE 评分法

用于产品功能优先级排序：

```
RICE Score = (Reach × Impact × Confidence) / Effort
```

| 维度           | 含义       | 评分标准                              |
| -------------- | ---------- | ------------------------------------- |
| **R**each      | 影响用户数 | 每季度影响的用户数量                  |
| **I**mpact     | 影响程度   | 3=重大, 2=高, 1=中, 0.5=低, 0.25=极低 |
| **C**onfidence | 信心程度   | 100%=高, 80%=中, 50%=低               |
| **E**ffort     | 工作量     | 人/月                                 |

**示例**：

| 功能       | Reach | Impact | Confidence | Effort | Score |
| ---------- | ----- | ------ | ---------- | ------ | ----- |
| 新登录流程 | 5000  | 2      | 80%        | 2      | 4000  |
| 性能优化   | 10000 | 1      | 50%        | 1      | 5000  |
| 新功能 X   | 2000  | 3      | 100%       | 4      | 1500  |

### 3. MoSCoW 优先级法

| 类别            | 含义   | 示例                       |
| --------------- | ------ | -------------------------- |
| **M**ust have   | 必须有 | 核心功能，缺少则产品不可用 |
| **S**hould have | 应该有 | 重要功能，但有替代方案     |
| **C**ould have  | 可以有 | 锦上添花，资源允许时做     |
| **W**on't have  | 不做   | 明确不在范围内             |

## 风险评估框架

### 1. 风险矩阵

```
        高概率
           │
    监控区  │  高风险区
    (中)   │  (高)
           │
低影响 ────┼──── 高影响
           │
    接受区  │  应对区
    (低)   │  (中)
           │
        低概率
```

### 2. 风险登记表

| 风险           | 概率 | 影响 | 等级 | 应对策略     | 负责人     |
| -------------- | ---- | ---- | ---- | ------------ | ---------- |
| 技术难点超预期 | 中   | 高   | 高   | 提前原型验证 | 技术负责人 |
| 资源不足       | 低   | 高   | 中   | 备选供应商   | 项目经理   |
| 需求变更       | 高   | 中   | 中   | 敏捷迭代     | 产品经理   |

### 3. 预期值分析

```
预期值 = 成功概率 × 成功收益 - 失败概率 × 失败损失
```

| 方案 | 成功率 | 成功收益 | 失败损失 | 预期值        |
| ---- | ------ | -------- | -------- | ------------- |
| A    | 70%    | 100万    | 30万     | 70-9 = 61万   |
| B    | 50%    | 200万    | 50万     | 100-25 = 75万 |
| C    | 90%    | 50万     | 10万     | 45-1 = 44万   |

## 可行性评估

### 1. TELOS 可行性分析

| 维度            | 问题             | 评估                     |
| --------------- | ---------------- | ------------------------ |
| **T**echnical   | 技术上可行吗？   | □ 可行 □ 有挑战 □ 不可行 |
| **E**conomic    | 经济上合理吗？   | □ 可行 □ 有挑战 □ 不可行 |
| **L**egal       | 法律上允许吗？   | □ 可行 □ 有挑战 □ 不可行 |
| **O**perational | 运营上可持续吗？ | □ 可行 □ 有挑战 □ 不可行 |
| **S**cheduling  | 时间上可达成吗？ | □ 可行 □ 有挑战 □ 不可行 |

## 决策框架

### 1. 六顶思考帽决策

依次从六个角度评估方案：

```
⬜ 白帽 - 客观事实
我们知道什么？数据说明什么？

🟥 红帽 - 情感直觉
直觉告诉我什么？感受如何？

⬛ 黑帽 - 批判谨慎
有什么风险？可能的问题？

🟨 黄帽 - 积极乐观
有什么好处？最好的情况？

🟩 绿帽 - 创意创新
还有其他方法吗？如何创新？

🟦 蓝帽 - 过程控制
下一步是什么？如何组织？
```

### 2. 决策树

```
                    ┌── 结果 A1 (预期收益)
          ┌─ 选项 A ─┤
          │        └── 结果 A2 (预期收益)
          │
起点 ────┼─ 选项 B ─── 结果 B1 (预期收益)
          │
          │        ┌── 结果 C1 (预期收益)
          └─ 选项 C ─┤
                    └── 结果 C2 (预期收益)
```

### 3. 假设对比法

| 假设         | 方案 A      | 方案 B      |
| ------------ | ----------- | ----------- |
| 如果市场增长 | ✅ 高收益   | ✅ 中收益   |
| 如果市场萎缩 | ❌ 亏损大   | ⚠️ 小幅亏损 |
| 如果竞争加剧 | ⚠️ 难以应对 | ✅ 有优势   |
| 如果技术变革 | ✅ 可适应   | ❌ 需重做   |

## 验证与复盘

### 假设验证清单

| 假设            | 验证方法 | 结果    | 下一步   |
| --------------- | -------- | ------- | -------- |
| 用户需要 X 功能 | 用户访谈 | ✅ 验证 | 继续开发 |
| 技术方案可行    | 原型测试 | ⚠️ 部分 | 需调整   |
| 成本在预算内    | 估算核算 | ❌ 失败 | 重新规划 |

### 复盘框架

**4L 复盘法**：

- **L**iked: 做得好的地方
- **L**earned: 学到的经验
- **L**acked: 缺少什么
- **L**onged for: 期望改进的

**5 Why 根因分析**：

```
问题: 项目延期

Why 1: 为什么延期？→ 开发时间不足
Why 2: 为什么时间不足？→ 需求变更多
Why 3: 为什么变更多？→ 需求不明确
Why 4: 为什么不明确？→ 前期沟通不足
Why 5: 为什么沟通不足？→ 没有需求确认流程

根因: 缺少需求确认流程
```
