# 输出模式

## 模板模式

提供输出格式模板，根据需求调整严格程度。

### 严格模板（API 响应、数据格式）

```markdown
## 报告结构

**必须**使用以下模板：

# [分析标题]

## 执行摘要

[一段话概述关键发现]

## 关键发现

- 发现 1：[描述] + [支撑数据]
- 发现 2：[描述] + [支撑数据]
- 发现 3：[描述] + [支撑数据]

## 建议

1. [具体可执行的建议]
2. [具体可执行的建议]
```

### 灵活模板（需要适应性调整）

```markdown
## 报告结构

以下是建议格式，可根据实际情况调整：

# [分析标题]

## 执行摘要

[概述]

## 关键发现

[根据发现内容调整章节]

## 建议

[针对具体上下文定制]
```

## 示例模式

当输出质量依赖于看到示例时，提供输入/输出对：

```markdown
## Commit 消息格式

按以下示例生成提交信息：

**示例 1：**
输入：添加了基于 JWT 的用户认证
输出：
feat(auth): 实现 JWT 认证机制

添加登录端点和令牌验证中间件

**示例 2：**
输入：修复了报告中日期显示错误的 bug
输出：
fix(reports): 修正时区转换中的日期格式化

统一使用 UTC 时间戳生成报告

遵循格式：type(scope): 简短描述，然后详细说明。
```

## 分级输出模式

根据严重程度分类输出：

```markdown
## 问题报告格式

按严重级别分类：

### P0 - 阻塞级（必须立即修复）

- [问题描述]
- [影响范围]
- [修复建议]

### P1 - 警告级（建议修复）

- [问题描述]
- [修复建议]

### P2 - 建议级（可选改进）

- [改进建议]
```

## 结构化数据模式

输出需要机器可读时：

```markdown
## 输出格式

输出 JSON 结构：

{
"summary": "一句话总结",
"findings": [
{
"type": "error|warning|info",
"location": "文件:行号",
"message": "问题描述",
"suggestion": "修复建议"
}
],
"metrics": {
"total_issues": 0,
"critical": 0,
"warnings": 0
}
}
```

## 交互式输出模式

需要用户确认或选择时：

```markdown
## 交互流程

1. 展示分析结果
2. 提供选项：
   - A: [选项描述]
   - B: [选项描述]
   - C: 自定义...
3. 等待用户选择
4. 根据选择继续
```
